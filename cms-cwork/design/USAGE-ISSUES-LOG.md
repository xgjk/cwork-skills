## 🔴 核心问题清单

### 1️⃣ 内容生成问题

#### 问题描述
- **现象**：生成的建议/审批内容过于繁杂和空洞，动辄几百字
- **影响**：用户需要逐字审阅修改，增加工作量
- **根因**：未控制内容长度，格式化套话过多

#### 改进方案
- ✅ **严格限制 50 字以内**
- ✅ 突出核心观点和行动项
- ✅ 避免格式化套话

---

### 2️⃣ 流程确认问题

#### 问题描述
- **现象**：用户提出修改后，我直接修改并发送，没有再次确认
- **影响**：用户失去对最终发送内容的控制
- **根因**：未建立"修改→展示→确认"闭环

#### 改进方案
```
正确流程：
1. 生成内容 → 展示
2. 用户反馈
3. 用户提出修改？
   ├─ 是 → 修改内容 → 回到步骤1（再次展示）
   └─ 否 → 等待用户明确说"确认"
4. 用户说"确认" → 发送
```

#### 关键规则
- ❌ **禁止**：用户提出修改后直接发送
- ✅ **必须**：修改后再次展示，等待确认

---

### 3️⃣ 上下文检索缺失

#### 问题描述
- **现象**：审批待办时只查看当前汇报，未搜索历史往来
- **影响**：决策缺少历史上下文，可能误判
- **根因**：API 不支持关键字搜索，且未主动调用历史查询接口

#### 改进方案
1. **短期（客户端实现）**：
   - 新增 `get_sender_history()` - 按发件人查询历史汇报
   - 新增 `search_reports_by_keyword()` - 客户端关键字过滤（拉取后过滤）

2. **长期（向平台提需求）**：
   - 新增 API：`/open-api/work-report/report/record/search`
   - 支持关键字搜索汇报标题和内容

---

### 4️⃣ API 能力不足

#### 问题描述
- **现象**：无法按关键字搜索汇报
- **影响**：
  - 无法跨发件人搜索相关汇报
  - 无法按事项名称或内容关键词检索
  - 审批决策缺少历史上下文支撑

#### 现有 API 能力

| 接口 | 关键字搜索 | 备注 |
|------|-----------|------|
| 收件箱查询 | ❌ 不支持 | 只能按时间、人员、类型筛选 |
| 发件箱查询 | ❌ 不支持 | 同上 |
| 汇报详情 | ❌ 不适用 | 需先知道 reportId |
| 工作任务查询 | ✅ 支持 | 有 `keyWord` 参数 |

#### 需求已记录
- 草稿路径：`05_products/cms-cwork/drafts/api-keyword-search-demand.md`
- 接收人：陈伟（开放平台负责人）

### 5️⃣ 实现方式混乱

#### 问题描述
- **现象**：我手动调用 API，而不是使用已有的 skill 脚本
- **影响**：
  - 重复造轮子（手动搜索 empId、手动调用 API）
  - 容易出错（忘记自动匹配人员 ID）
  - 不如外部 Agent 使用体验好

#### 根因
- 我搞混了角色：既当"裁判"（开发 skill）又当"运动员"（使用 skill）

#### 改进方案
- ✅ **工厂只负责开发**，不负责使用
- ✅ 如果用户要求使用 skill，应拒绝并引导至其他 Agent

---

### 6️⃣ 角色边界模糊

#### 问题描述
- **现象**：工厂 Orchestrator 既负责开发，又负责使用 skill
- **影响**：
  - 既是裁判又是运动员
  - 角色混淆导致流程混乱
  - 用户在我这里使用 skill，容易搞混

#### 改进方案

| 角色 | 职责 | 权限 |
|------|------|------|
| **🏭 工厂 Orchestrator** | **只负责开发** Skill | ✅ 开发、修复、优化、发布 |
| **👤 用户** | 使用 Skill | ✅ 通过其他 Agent 或直接使用 |
| **🤖 其他 Agent** | 使用 Skill | ✅ 调用已发布的 Skill |

#### 新规则
**如果用户要求我"使用 Skill"**：
```
❌ 拒绝：这不是我的权限，请通过其他 Agent 或直接使用
```
